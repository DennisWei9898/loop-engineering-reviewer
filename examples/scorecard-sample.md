# 審查計分卡 — 範例（示範用）

> 這是對一個假想專案「每日自動產報告的 cron agent」跑完一輪審查後的計分卡長相。重點看三件事：**逐條 PASS/FAIL/WARN + 原文證據**、**八二法則的提問分流**（先問影響最大的 3 個，不是 50 題轟炸）、以及**只修合規、不碰風格**的修正提案。

---

## 受審對象

`daily-report-agent`（cron 每日 06:00 觸發 → 抓數據 → 寫報告 → 自動寄出）

## 計分卡：3/7

| # | 量尺 | 判定 | 證據（原文） |
|---|---|---|---|
| L1 | Maker/Verifier 分離 | ❌ FAIL | 「產完報告後自己檢查一遍格式沒問題就寄出」——寫的人自己驗收 |
| L2 | Gate 客觀可判 | ❌ FAIL | 過閘條件是「內容看起來合理」，無機器可判訊號 |
| L3 | 最小可行迴圈四零件 | ⚠️ WARN | 有自動化、有 skill，但無 state file——失敗當天無紀錄可回溯 |
| L4 | 停止點 + 迭代上限 | ✅ PASS | 「重試最多 2 次，仍失敗則跳過本日」 |
| L5 | 不可逆前人工閘 | ❌ FAIL | 報告**自動寄給客戶**，寄出前沒有任何人工確認——對外發布屬不可逆 |
| L6 | 成功指標＝採納率 | ⚠️ WARN | 只記錄「有沒有寄出」，不記錄收件人是否採用/退回 |
| L7 | 防三大陷阱 | ❌ FAIL | 抓數失敗時 fallback 用前一日數據且**不標註**——教科書級無聲失敗 |

## 八二分流：先問這 3 題（其餘先不煩你）

1. **L5（最高風險）**：寄給客戶前要插一道人工確認，還是先改成寄到你自己的信箱過渡？
2. **L7（真 bug）**：fallback 用舊數據可以，但必須在報告上標註「數據為 MM/DD 快照」——可以直接這樣修嗎？
3. **L1**：驗收改由獨立 fresh-context agent 對照 checklist 判定，checklist 我從你現有的「格式檢查」擴寫——要嗎？

（L2/L3/L6 屬結構補強，第二批處理；本輪先把 3/7 拉到 5/7。）

## 修正原則聲明

以上修正**只動迴圈結構**：你的報告文風、章節安排、用詞習慣一律不碰。
